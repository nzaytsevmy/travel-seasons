# Quality Radar

Не путать с обычным PR-ревью или пиксельной визуальной регрессией — это **интерактивный
аудит живого прода**, единственный механизм, который реально кликает по сайту traveltribe.ru
так, как это делает читатель, а не просто сверяет HTML/скриншоты сборки.

## Скоуп — что проверяем (уникальная ценность, не дублируем гейты)

1. **Интерактив живого прода.** Клики по фильтрам (`/packing/*/*/`, главная страница
   месяц/бюджет/вайб, `/compare/`, калькулятор) — и **ассерт видимости после клика**
   (`offsetParent !== null` / `getComputedStyle(...).display`), не просто наличие
   элемента в DOM. Статический скриншот сборки такое не ловит: элемент физически
   присутствует в HTML, но CSS может не давать ему исчезнуть.
2. **Рендер-геометрия на живом сайте** @360 (мобайл) и @1280 (десктоп): touch-таргеты
   ≥44px, overflow, currency `&nbsp;`-перенос, задвоенные глифы («от от»), битые
   изображения. Именно на **проде**, не на dist/ или localhost — прод может отличаться
   от сборки (кэш CDN, race условия деплоя, устаревшая версия на хостинге).
3. **Консистентность чисел контента.** Захардкоженные величины в тексте/title/schema
   («70 направлений») против реальных данных сайта (`DIRECTIONS.length` и т.п.) —
   расхождение = путаный сигнал для читателя и поисковика.
4. **Подача монетизационных CTA.** Не факт наличия `erid`/`sponsored` (это уже ловит
   content-invariants) — а визуальное поведение: контраст, hover/focus, не съезжает
   ли при переносе строк.

## Анти-скоуп — что НЕ проверяем (уже покрыто, дублировать = чистый шум)

| Класс находок | Кто уже проверяет |
|---|---|
| canonical/sitemap/OG/dateModified/orphan/generic-анкоры | `npm run check:seo` (`scripts/seo-site-audit.mjs`), встроен в pre-push |
| Метка «реклама», утечка имён файлов-исходников, жаргон, хвостовая «→», пост без `sponsored` | `tests/content-invariants.spec.ts` (9 инвариантов, гоняется на dist) |
| CWV (LCP/CLS/TBT/INP) desktop+mobile | `.github/workflows/lighthouse.yml` (на каждый PR) |
| Пиксельная визуальная регрессия | `.github/workflows/visual-tests.yml` (на каждый PR) |
| Freshness (правка контента без бампа `DATA_UPDATED`) | `.github/workflows/build.yml` (freshness-gate) |
| SEO/CTR-возможности: запросы поз 6-15 (striking distance), показы много/кликов нет (zero-click) | `.github/scripts/seo_pulse.py` — модули B1/B2, еженедельный отчёт SEO Pulse |

Находка попадает в анти-скоуп → **не репортить**, это дублирование существующего гейта,
не работа Radar. Если сомневаешься, попадает ли находка в анти-скоуп — грепни репо на
предмет, не ловит ли её уже один из перечисленных файлов, прежде чем включать в отчёт.

## Процедура прогона

1. **Дедуп сначала.** Прочитать `radar/findings.json`. Если отчёт этой недели
   (`radar/RADAR-<год>-W<неделя>.md`) уже есть в main и main не менялся с момента его
   генерации — полный краул дал бы идентичный результат. Вместо повторного прогона:
   точечно перепроверить живьём открытые (`status: open`) находки реестра, дописать
   аддендум в существующий файл, не создавать дубликат.
2. **Live-краул прода** (`https://traveltribe.ru`, не localhost/dist) через Playwright,
   @360 и @1280:
   - Триггерить интерактив из раздела «Скоуп» выше — клик по каждому классу фильтров,
     затем ассерт видимости результата.
   - Скрин-доказательство для каждой P1/P2 находки → `radar/assets/<id>.png`.
3. **Канон-критик**: сверка со списком запрещённых паттернов в `CLAUDE.md`
   («Design canon» → «НЕ использовать») и `BRANDBOOK.md` — декоративные эмодзи,
   стрелки в CTA-тексте, eyebrow-капс, скругления >12px на текстовых секциях,
   generic cards-grid с тенями.
4. **Консистентность чисел**: grep захардкоженных величин по репо/`dist/`, сверка
   с исходными данными (`src/data/*.js`).
5. **Приоритизация**: 🔴 P1 (сломана функциональность, юзер реально видит баг) /
   🟠 P2 (заметное расхождение, use case не ломает) / 🟡 P3 (косметика).
6. **Findings-registry** (`radar/findings.json`, массив объектов):
   ```json
   {"id": "packing-hidden-filter", "severity": "P1", "url": "https://traveltribe.ru/packing/egypt/june/",
    "file": "src/components/PackingChecklist.astro", "status": "open",
    "first_seen": "2026-06-29", "last_seen": "2026-07-14", "note": "..."}
   ```
   `id` — стабильный человекочитаемый слаг (не хэш — так его проще узнать глазами при
   диффе), не меняется между прогонами для той же находки. Перед репортом: перечитать
   реестр, для существующих `open` — перепроверить живьём (закрылась? → `fixed`,
   владелец решил не чинить? → `wontfix`, не трогать эти статусы без внешнего сигнала).
   `fixed`/`wontfix` **не репортить повторно** в Telegram/отчёте — только держать в
   реестре для истории. Новые находки — добавлять с `status: open`.
7. **Отчёт** — писать/обновлять `radar/RADAR-<год>-W<неделя>.md` по формату ниже.
8. **Коммит прямо в main** (см. «Правила исполнения» — НЕ PR-per-run). Telegram — только
   дельта прогона (новые находки + находки, сменившие статус с прошлого раза); если
   дельты нет — не слать (анти-спам).

## Формат отчёта

```markdown
# 📡 Quality Radar W<неделя>-<год>

Еженедельный интерактивный аудит прода traveltribe.ru: живой Playwright @360/@1280
+ канон-критик + консистентность чисел. Анти-скоуп (SEO-техника/CWV/пиксели/контент-
инварианты) — см. `radar/SKILL.md`. Только отчёт, код не менялся.

## Топ находок

**🔴 P1 — <заголовок>**
<что сломано, конкретный URL, `file:line`, скрин в `radar/assets/`, предлагаемый фикс
одной-двумя строками>

**🟠 P2 — <...>**

**🟡 P3:** <короткий список, без разворачивания>

## Проверено и чисто

<что прогнали и не нашли проблем — список коротко, чтобы было видно объём покрытия>

## Needs-review

<неуверенные находки, где нужно решение владельца, не однозначный баг>
```

## Правила исполнения (жёсткие — из граблей прогона W27, 2026-06-29/30)

- **НИКОГДА `git checkout`/`switch` в `~/seasons-work`.** Это общий рабочий каталог —
  параллельная сессия может одновременно переключать ветку и коммитить. Работать
  только через GitHub API (`create_branch` + `create_or_update_file`/PUT contents)
  или изолированный `git worktree`.
- **Коммитить отчёт и `findings.json` ПРЯМО в main**, не открывать PR-per-run. Урок:
  PR #41 провисел open 2 недели без ревью, находки протухли, P1-баг не чинился —
  PR-путь для еженедельного авто-отчёта = гарантированный PR-rot. Push — с ретраем
  (rebase + до 3 попыток), как в `seo-pulse.yml` — гонка с другими коммитами реальна.
- **Авто-фикс запрещён** для контента/денег/schema/логики компонентов — Radar только
  предлагает фикс текстом в отчёте, не коммитит правку кода сам. (Раньше обсуждался
  узкий разрешённый класс auto-fix для wrap/overflow/double-glyph — не включён в этот
  прогон; если понадобится, делать отдельным явным шагом с ревью Никиты, не молча.)
- **Telegram — анти-спам**: слать только при реальной дельте с прошлого прогона.
- **Не выдумывать находки ради отчёта.** Пустой раздел «Топ находок» — нормальный,
  честный результат недели без регрессий. Секция «Проверено и чисто» существует
  именно для этого — показать объём покрытия, не наполнять «находки» искусственно.
